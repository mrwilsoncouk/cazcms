# Performance features added

1. Static-first pages are generated by `npm run build` into `public/site`.
2. Internal cache endpoints live under `/api/performance/cache`.
3. Build cache data tables and route manifests are included.
4. Media records support optimized variants; static HTML uses lazy images.
5. HTML/CSS/JS are minified during static build.
6. Critical CSS is inlined and non-critical CSS is deferred.
7. Public pages use tiny island JavaScript only for interactive pieces.
8. `public/site/routes.manifest.json` is generated for fast route lookup.
9. Query index definitions are stored in `data/queryIndexes.json`.
10. API list endpoints should use pagination parameters; new performance guidance is coded into docs and modules.
11. Background jobs are recorded for static builds, precomputation, compression and indexing.
12. Brotli and gzip files are generated, and the HTTP layer serves them when supported.
13. Hashed assets receive long immutable cache headers; HTML/API gets shorter headers.
14. Admin remains served separately from the public static frontend.
15. Navigation, breadcrumbs and related product data are precomputed.
16. Local search index is generated into `data/searchIndexes.json`.
17. Images, comments and related widgets are lazy-loaded.
18. Admin bundle split points are documented for content, commerce, users, analytics and settings.
19. Railway production env examples are included below.
20. Performance budgets are checked by `/api/performance/budget` and during build.

Recommended Railway variables:

```bash
NODE_ENV=production
API_CACHE_TTL_MS=30000
PERF_MAX_JS_BYTES=180000
PERF_MAX_IMAGE_BYTES=450000
PERF_MAX_PAGE_BYTES=250000
PERF_MAX_API_MS=300
RAILWAY_REGION=your-nearest-region
DATA_DIR=/app/data
```
